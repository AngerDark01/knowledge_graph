### 添加新布局算法

1. 实现 `ILayoutAlgorithm` 接口
2. 在 `src/services/layout/algorithms/` 创建新的算法类
3. 定义算法逻辑和配置参数
4. 在策略中使用新算法

**示例 - 添加圆形布局算法**:
```typescript
// src/services/layout/algorithms/CircularAlgorithm.ts
import { ILayoutAlgorithm, LayoutAlgorithmOptions } from './ILayoutAlgorithm';
import { Node, Group } from '../../../types/graph/models';

export class CircularAlgorithm implements ILayoutAlgorithm {
  readonly name = 'Circular Layout';
  readonly id = 'circular';

  calculate(nodes: (Node | Group)[], options?: LayoutAlgorithmOptions): (Node | Group)[] {
    if (nodes.length === 0) return [];

    const centerX = options?.centerX ?? 0;
    const centerY = options?.centerY ?? 0;
    const radius = options?.radius ?? 200;

    return nodes.map((node, index) => {
      const angle = (2 * Math.PI * index) / nodes.length;
      const x = centerX + radius * Math.cos(angle);
      const y = centerY + radius * Math.sin(angle);

      return {
        ...node,
        position: { x, y }
      };
    });
  }
}
```

2. 在策略中使用算法：
```typescript
// 在 CanvasLayoutStrategy 或其他策略中
const circularAlgorithm = new CircularAlgorithm();
const layoutedNodes = circularAlgorithm.calculate(nodes, options);
```

### 性能优化建议

1. **使用 memo 包装组件**
   - 所有节点和边组件都使用 `React.memo`
   - 减少不必要的重新渲染

2. **避免频繁的 Store 更新**
   - 拖拽时使用 `isDraggingRef` 阻止同步
   - 位置/尺寸变化不添加历史记录

3. **大规模节点处理**
   - 使用 Web Workers 进行密集计算
   - 实现分批处理，避免阻塞 UI 线程
   - 使用四叉树等空间分区算法优化碰撞检测

4. **使用 useMemo 和 useCallback**
   - 缓存计算结果
   - 稳定函数引用

5. **布局算法优化**
   - 使用节点映射以提高查找性能
   - 限制迭代次数防止无限循环
   - 多层嵌套时使用递归算法但设置深度限制

## 常见问题

### 1. 节点拖拽时闪烁
原因：Store 同步与拖拽冲突
解决：使用 `isDraggingRef` 在拖拽时阻止同步

### 2. 群组边界不更新
原因：没有调用 `updateGroupBoundary`
解决：在添加/移动节点后调用此函数

### 3. 历史记录太多导致内存问题
原因：`maxSize` 设置过大
解决：调整 `historySlice.ts` 中的 `maxSize`（默认 50）

### 4. 节点超出群组边界
原因：位置约束逻辑失效
解决：检查 `constrainNodeToGroupBoundary` 函数

### 5. 跨群组边样式不正确
原因：`isCrossGroup` 标记未设置
解决：在 `onConnect` 中正确检测和设置

### 6. 布局时子节点位置错乱
原因：多层嵌套结构下，父节点移动时没有维持所有嵌套子节点的相对位置
解决：
   - 检查 `CanvasLayoutStrategy.ts` 中的 `updateNestedNodePositionsForCanvasLayout` 方法
   - 确保使用 `NestedNodePositionUpdater` 正确处理嵌套节点位置更新
   - 验证 `CoordinateTransformer.relativeToAbsoluteForGroup` 方法

### 7. 嵌套群组布局结果不正确
原因：递归布局策略未能正确处理多层嵌套
解决：
   - 检查 `RecursiveLayoutStrategy.ts` 的执行流程
   - 确保从最深层到最浅层的布局顺序
   - 验证群组大小调整是否在正确时机执行

### 8. 布局性能问题
原因：节点数量过多时算法效率低
解决：
   - 使用四叉树优化碰撞检测算法
   - 实现分页布局处理大量节点
   - 考虑使用 Web Workers 进行密集计算

### 9. 布局结果不稳定
原因：算法中存在随机因素或浮点计算误差
解决：
   - 确保算法逻辑确定性
   - 使用整数位置值避免浮点误差

### 10. 群组大小调整不当
原因：GroupSizeAdjuster 计算边界有误
解决：
   - 检查 `calculateChildrenRelativeBounds` 方法
   - 验证 padding 和 margin 的计算逻辑
   - 确保最小尺寸限制正确应用

## 最佳实践

### 布局算法开发
1. **遵循算法接口**：始终实现 `ILayoutAlgorithm` 接口
2. **纯函数设计**：算法应仅依赖输入参数，不修改全局状态
3. **参数化设计**：通过选项对象配置算法行为
4. **性能考虑**：考虑算法的时间复杂度，特别是对大规模节点的支持

### 策略开发
1. **职责单一**：每个策略只处理一种布局场景
2. **组合复用**：合理使用算法层和工具层
3. **错误处理**：妥善处理异常情况
4. **统计信息**：提供详细的执行统计

### 工具函数开发
1. **无副作用**：工具函数应该是纯函数
2. **可复用**：设计为可被多个策略使用
3. **性能优化**：考虑大数据集下的性能表现
4. **充分测试**：为边界情况编写测试用例
