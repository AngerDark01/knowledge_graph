## 🎯 工具层职责

工具层提供纯函数式的工具类，具有以下特点：

1. **无副作用**：不修改输入参数，总是返回新对象
2. **可独立使用**：不依赖其他层，可以单独测试
3. **高度复用**：被策略层和算法层广泛使用
4. **职责单一**：每个工具类只负责一个明确的功能

## 1️⃣ CoordinateTransformer - 坐标转换工具

### 核心职责

处理不同坐标系之间的转换，确保节点位置计算的正确性。

### 为什么需要坐标转换？

在布局系统中存在多个坐标系：

1. **相对坐标系**：算法计算的相对位置（相对于 0,0 点）
2. **群组内坐标系**：相对于群组左上角的位置
3. **画布绝对坐标系**：相对于画布原点的位置
4. **第四象限坐标系**：所有坐标都是正数的坐标系

不同的布局场景需要在这些坐标系之间转换。

### 主要方法详解

#### 方法1：`toAbsolute()` - 转换为绝对坐标

**功能**：将相对坐标转换为画布绝对坐标

**输入**：
- `nodes`：使用相对坐标的节点列表
- `anchor`：锚点位置（基准点）

**处理逻辑**：
1. 遍历每个节点
2. 将节点的相对位置加上锚点位置
3. 返回新的节点列表（不修改原数组）

**应用场景**：
- 群组内布局：将群组内的相对位置转换为画布绝对位置
- 画布布局：将居中布局的相对位置转换为绝对位置

**实现原理**：
```
绝对位置 = 相对位置 + 锚点位置

例如：
- 节点相对位置：(50, 30)
- 锚点位置：(100, 200)
- 结果绝对位置：(150, 230)
```

---

#### 方法2：`toQuadrantFour()` - 转换到第四象限

**功能**：确保所有节点位置都是正数（第四象限坐标系）

**输入**：
- `nodes`：可能包含负坐标的节点列表

**处理逻辑**：
1. 找出所有节点中最小的 x 和 y 坐标
2. 如果最小值小于 0，计算偏移量
3. 将所有节点向正方向平移
4. 确保最左上角的节点位于 (0, 0) 附近

**应用场景**：
- GridAlgorithm 使用，确保从 (0,0) 开始布局
- 避免负坐标导致的显示问题

**实现原理**：
```
找到最小坐标：minX, minY
如果 minX < 0 或 minY < 0：
  offsetX = -minX
  offsetY = -minY
  每个节点位置 += (offsetX, offsetY)
```

---

#### 方法3：`calculateGroupAnchor()` - 计算群组锚点

**功能**：计算群组的布局锚点位置

**输入**：
- `group`：群组对象（包含位置、padding等信息）

**处理逻辑**：
1. 获取群组的左上角位置
2. 加上 paddingLeft 和 paddingTop
3. 得到群组内容区的左上角位置

**应用场景**：
- 群组内布局：确定子节点布局的起始位置
- 保持固定锚点策略

**实现原理**：
```
锚点 X = 群组 X + paddingLeft
锚点 Y = 群组 Y + paddingTop

例如：
- 群组位置：(100, 200)
- padding：{ left: 20, top: 20 }
- 锚点位置：(120, 220)
```

**关键设计**：使用固定锚点（左上角+padding），而非中心点，避免锚点漂移问题。

---

#### 方法4：`relativeToAbsoluteForGroup()` - 群组专用转换

**功能**：将相对布局直接转换为群组内的绝对位置

**输入**：
- `nodes`：使用相对坐标的节点列表（从 0,0 开始）
- `parentGroup`：父群组对象

**处理逻辑**：
1. 调用 `calculateGroupAnchor()` 计算锚点
2. 调用 `toAbsolute()` 转换为绝对坐标
3. 一步到位完成转换

**应用场景**：
- GroupLayoutStrategy 中最常用
- 将 GridAlgorithm 的结果转换为画布坐标

**实现原理**：
```
步骤1：计算群组锚点
  anchor = calculateGroupAnchor(parentGroup)

步骤2：转换为绝对坐标
  absoluteNodes = toAbsolute(nodes, anchor)
```

---

### 使用示例

**场景：群组内布局**

```
1. GridAlgorithm 计算相对布局
   节点A: (0, 0)
   节点B: (150, 0)
   节点C: (0, 100)

2. 群组位置和 padding
   群组位置: (500, 300)
   padding: { left: 20, top: 20 }

3. 计算锚点
   锚点 = (500 + 20, 300 + 20) = (520, 320)

4. 转换为绝对坐标
   节点A: (520, 320)
   节点B: (670, 320)
   节点C: (520, 420)
```

## 2️⃣ GroupSizeAdjuster - 群组大小调整工具

### 核心职责

自动调整群组的宽度和高度，确保群组能够完整包含所有子节点。

### 为什么需要自动调整？

布局后子节点的位置可能超出群组边界，需要：
1. 扩大群组以包含所有子节点
2. 保持合适的 padding
3. 更新群组的 boundary 信息

### 主要方法详解

#### 方法1：`adjustSingleGroup()` - 调整单个群组

**功能**：调整指定群组的大小以适应其子节点

**输入**：
- `groupId`：要调整的群组 ID
- `allNodes`：所有节点列表（包括群组和子节点）

**处理逻辑**：

**步骤1：找到目标群组**
- 在节点列表中找到指定 ID 的群组
- 如果找不到，返回原节点列表

**步骤2：获取直接子节点**
- 筛选出 groupId 等于目标群组 ID 的节点
- 这些是群组的直接子节点（不包括孙节点）

**步骤3：计算子节点边界**
- 调用 `calculateChildrenRelativeBounds()` 计算子节点的相对边界
- 得到子节点区域的最小/最大 x/y 坐标

**步骤4：计算新的群组尺寸**
```
新宽度 = 子节点宽度 + paddingLeft + paddingRight + 额外边距
新高度 = 子节点高度 + paddingTop + paddingBottom + 额外边距
```

**步骤5：更新群组对象**
- 更新 width 和 height
- 更新 boundary（左、右、上、下边界）

**应用场景**：
- GroupLayoutStrategy 布局完成后
- RecursiveLayoutStrategy 每层布局后

**关键参数**：
- `MIN_GROUP_SIZE`：群组最小尺寸（400 x 300）
- `EXTRA_MARGIN`：额外边距（20px）

---

#### 方法2：`adjustMultipleGroups()` - 批量调整群组

**功能**：按顺序调整多个群组的大小

**输入**：
- `groupIds`：要调整的群组 ID 列表
- `allNodes`：所有节点列表

**处理逻辑**：
1. 遍历群组 ID 列表
2. 对每个群组调用 `adjustSingleGroup()`
3. 使用上一次调整的结果作为下一次的输入

**应用场景**：
- RecursiveLayoutStrategy 中从深到浅调整所有群组

**实现原理**：
```
let workingNodes = allNodes
for (groupId of groupIds) {
  workingNodes = adjustSingleGroup(groupId, workingNodes)
}
return workingNodes
```

---

#### 方法3：`calculateChildrenRelativeBounds()` - 计算子节点边界

**功能**：计算子节点相对于父群组的边界范围

**输入**：
- `children`：子节点列表
- `parentGroup`：父群组对象

**处理逻辑**：

**步骤1：计算每个子节点的四个边界**
```
对于每个子节点：
  左边界 = 节点X - 节点宽度/2 - 群组X
  右边界 = 节点X + 节点宽度/2 - 群组X
  上边界 = 节点Y - 节点高度/2 - 群组Y
  下边界 = 节点Y + 节点高度/2 - 群组Y
```

**步骤2：找出整体边界**
```
minX = min(所有左边界)
maxX = max(所有右边界)
minY = min(所有上边界)
maxY = max(所有下边界)
```

**步骤3：计算相对尺寸**
```
relativeWidth = maxX - minX
relativeHeight = maxY - minY
```

**返回值**：
```javascript
{
  minX,     // 最左边的相对位置
  maxX,     // 最右边的相对位置
  minY,     // 最上面的相对位置
  maxY,     // 最下面的相对位置
  width,    // 子节点区域宽度
  height    // 子节点区域高度
}
```

**应用场景**：
- 被 `adjustSingleGroup()` 调用
- 计算群组需要的最小尺寸

---

### 使用示例

**场景：调整群组大小**

```
初始状态：
  群组位置: (100, 100)
  群组尺寸: 300 x 200
  padding: { left: 20, top: 20, right: 20, bottom: 20 }

子节点布局后：
  节点A: (120, 120) - 尺寸 80 x 60
  节点B: (250, 120) - 尺寸 80 x 60
  节点C: (120, 240) - 尺寸 80 x 60

计算边界：
  minX = 120 - 40 - 100 = -20
  maxX = 250 + 40 - 100 = 190
  minY = 120 - 30 - 100 = -10
  maxY = 240 + 30 - 100 = 170

  子节点宽度 = 190 - (-20) = 210
  子节点高度 = 170 - (-10) = 180

计算新群组尺寸：
  新宽度 = 210 + 20 + 20 + 20 = 270
  新高度 = 180 + 20 + 20 + 20 = 240

调整后：
  群组尺寸: 270 x 240
```

## 3️⃣ CollisionResolver - 碰撞检测与解决工具

### 核心职责

检测节点之间的重叠，并通过迭代调整解决碰撞问题。

### 为什么需要碰撞解决？

布局算法可能产生节点重叠的情况：
1. 节点尺寸不统一
2. 网格计算不够精确
3. 权重排序导致密集排列

碰撞解决确保最终布局中节点之间有足够的间距。

### 主要方法详解

#### 方法1：`resolve()` - 解决碰撞

**功能**：迭代调整节点位置直到没有碰撞或达到最大迭代次数

**输入**：
- `nodes`：可能存在碰撞的节点列表

**处理逻辑**：

**步骤1：初始化**
```
let workingNodes = nodes 的副本
maxIterations = 50（最多迭代50次）
minSeparation = 10px（最小间距）
```

**步骤2：迭代解决**
```
for (i = 0; i < maxIterations; i++) {
  if (!hasCollisions(workingNodes)) {
    break  // 没有碰撞，退出循环
  }

  workingNodes = 解决一次碰撞(workingNodes)
}
```

**步骤3：碰撞检测**
- 检查每对节点是否重叠
- 计算两个节点中心的距离
- 计算两个节点半径之和
- 如果距离 < 半径和 + minSeparation，则发生碰撞

**步骤4：碰撞解决策略**

使用**最小平移向量（MTV）**方法：
```
对于每对碰撞的节点A和B：

1. 计算重叠距离
   overlap = (radiusA + radiusB + minSeparation) - distance

2. 计算移动方向（从A指向B的单位向量）
   directionX = (B.x - A.x) / distance
   directionY = (B.y - A.y) / distance

3. 计算排斥力（两个节点各移动一半）
   repulsionX = directionX * overlap / 2
   repulsionY = directionY * overlap / 2

4. 应用排斥力
   A.x -= repulsionX
   A.y -= repulsionY
   B.x += repulsionX
   B.y += repulsionY
```

**返回值**：
- 解决碰撞后的节点列表

**应用场景**：
- 所有策略布局后
- 确保节点之间有足够间距

---

#### 方法2：`countCollisions()` - 统计碰撞数量

**功能**：统计当前有多少对节点发生碰撞

**输入**：
- `nodes`：节点列表

**处理逻辑**：
1. 双重循环检查每对节点
2. 对于每对节点，检查是否重叠
3. 累计碰撞对数

**返回值**：
- 碰撞对数（整数）

**应用场景**：
- 策略完成后报告统计信息
- 判断是否需要继续迭代

---

#### 方法3：`hasCollisions()` - 检测是否存在碰撞

**功能**：快速检测是否存在任何碰撞

**输入**：
- `nodes`：节点列表

**处理逻辑**：
1. 双重循环检查每对节点
2. 一旦发现碰撞立即返回 true
3. 没有碰撞返回 false

**返回值**：
- boolean（是否存在碰撞）

**应用场景**：
- resolve() 方法中的迭代终止条件

---

### 碰撞检测算法详解

#### 判断两个节点是否碰撞

**输入**：节点A和节点B

**步骤1：计算中心距离**
```
dx = B.x - A.x
dy = B.y - A.y
distance = sqrt(dx² + dy²)
```

**步骤2：计算半径之和**
```
radiusA = sqrt((A.width/2)² + (A.height/2)²)
radiusB = sqrt((B.width/2)² + (B.height/2)²)
requiredDistance = radiusA + radiusB + minSeparation
```

**步骤3：判断**
```
if (distance < requiredDistance) {
  发生碰撞
} else {
  没有碰撞
}
```

**为什么使用半径而非矩形碰撞检测？**
- 更平滑的排斥效果
- 避免矩形角落的锯齿问题
- 计算简单高效

---

### 使用示例

**场景：解决两个节点碰撞**

```
初始状态：
  节点A: (100, 100) - 尺寸 80 x 60
  节点B: (120, 105) - 尺寸 80 x 60

步骤1：检测碰撞
  距离 = sqrt((120-100)² + (105-100)²) = sqrt(425) ≈ 20.6
  半径A = sqrt(40² + 30²) = 50
  半径B = sqrt(40² + 30²) = 50
  需要距离 = 50 + 50 + 10 = 110
  20.6 < 110，发生碰撞！

步骤2：计算排斥力
  重叠 = 110 - 20.6 = 89.4
  方向X = (120-100)/20.6 ≈ 0.97
  方向Y = (105-100)/20.6 ≈ 0.24
  排斥X = 0.97 * 89.4 / 2 ≈ 43.4
  排斥Y = 0.24 * 89.4 / 2 ≈ 10.7

步骤3：应用排斥
  节点A: (100 - 43.4, 100 - 10.7) = (56.6, 89.3)
  节点B: (120 + 43.4, 105 + 10.7) = (163.4, 115.7)

结果：
  新距离 = sqrt((163.4-56.6)² + (115.7-89.3)²) ≈ 110
  碰撞解决！
```

## 4️⃣ NestingTreeBuilder - 嵌套树构建器

### 核心职责

构建和分析节点的嵌套层级关系，支持递归布局。

### 为什么需要嵌套树？

在递归布局中需要：
1. 知道有多少层嵌套
2. 知道每层有哪些群组
3. 从最深层向上逐层布局

### 主要方法详解

#### 方法1：`groupNodesByDepth()` - 按深度分组节点

**功能**：将所有节点按照嵌套深度分组

**输入**：
- `nodes`：所有节点列表

**处理逻辑**：

**步骤1：计算每个节点的深度**
```
对于每个节点：
  if (节点没有 groupId) {
    深度 = 0（顶层节点）
  } else {
    深度 = 递归计算父节点深度 + 1
  }
```

**步骤2：按深度分组**
```
创建 Map: depth -> nodes[]
遍历每个节点，将其加入对应深度的数组
```

**返回值**：
```javascript
Map<depth, nodes[]>
例如：
{
  0: [nodeA, nodeB, groupX],      // 顶层节点
  1: [nodeC, nodeD, groupY],      // 第一层嵌套
  2: [nodeE, nodeF]                // 第二层嵌套
}
```

**应用场景**：
- RecursiveLayoutStrategy 开始时
- 分析嵌套结构

---

#### 方法2：`getMaxDepth()` - 获取最大深度

**功能**：获取嵌套结构的最大深度

**输入**：
- `depthGroups`：深度分组 Map

**处理逻辑**：
```
找到 Map 中最大的 key 值
```

**返回值**：
- 最大深度（整数）

**应用场景**：
- 确定递归布局需要迭代多少层

---

#### 方法3：`getGroupsAtDepth()` - 获取指定深度的群组

**功能**：获取指定深度的所有群组节点

**输入**：
- `depthGroups`：深度分组 Map
- `depth`：目标深度

**处理逻辑**：
```
1. 从 Map 中获取指定深度的节点列表
2. 筛选出类型为 'group' 的节点
3. 返回群组列表
```

**返回值**：
- 群组节点列表

**应用场景**：
- RecursiveLayoutStrategy 逐层布局群组

---

#### 方法4：`calculateNodeDepth()` - 计算单个节点深度

**功能**：递归计算单个节点的嵌套深度

**输入**：
- `node`：要计算深度的节点
- `allNodes`：所有节点列表（用于查找父节点）

**处理逻辑**：
```
function calculateDepth(node):
  if (node 没有 groupId):
    return 0

  parentGroup = 查找 groupId 对应的群组
  if (找不到父群组):
    return 0

  return calculateDepth(parentGroup) + 1
```

**返回值**：
- 深度（整数，0表示顶层）

**应用场景**：
- `groupNodesByDepth()` 内部调用

---

### 使用示例

**场景：分析嵌套结构**

```
节点结构：
  GroupA (深度0, 顶层群组)
    ├─ NodeA1 (深度1)
    └─ GroupB (深度1, 嵌套群组)
        ├─ NodeB1 (深度2)
        └─ NodeB2 (深度2)

  GroupC (深度0, 顶层群组)
    └─ NodeC1 (深度1)

按深度分组结果：
  depth 0: [GroupA, GroupC]
  depth 1: [NodeA1, GroupB, NodeC1]
  depth 2: [NodeB1, NodeB2]

最大深度：2

指定深度的群组：
  getGroupsAtDepth(depthGroups, 0) -> [GroupA, GroupC]
  getGroupsAtDepth(depthGroups, 1) -> [GroupB]
  getGroupsAtDepth(depthGroups, 2) -> []
```

## 5️⃣ GeometryUtils - 几何计算工具

### 核心职责

提供各种几何计算功能，如边界计算、距离计算等。

### 主要功能

**边界计算**：
- 计算节点的边界框
- 计算多个节点的整体边界

**距离计算**：
- 计算两点之间的距离
- 计算点到矩形的最短距离

**碰撞检测**：
- 检测两个矩形是否重叠
- 检测点是否在矩形内

**其他工具**：
- 向量归一化
- 角度计算

这些功能被其他工具类和算法广泛使用。

## 🎯 工具层设计原则

### 1. 纯函数设计

所有工具方法都是纯函数：
- 不修改输入参数
- 相同输入总是产生相同输出
- 没有副作用

**好处**：
- 易于测试
- 易于理解
- 可以安全并发调用

### 2. 静态方法优先

大多数工具类使用静态方法：
- 不需要创建实例
- 降低内存开销
- 使用更简洁

**例外**：CollisionResolver 是实例方法，因为需要维护配置状态。

### 3. 职责单一

每个工具类只负责一个领域：
- CoordinateTransformer 只做坐标转换
- GroupSizeAdjuster 只做尺寸调整
- CollisionResolver 只做碰撞处理

### 4. 高度复用

工具层被多个上层模块使用：
- 策略层的所有策略都使用工具层
- 算法层的某些算法也使用工具层
- 可以独立导出供外部使用

## 🧪 测试建议

### 单元测试重点

**CoordinateTransformer**：
- 测试各种坐标转换的正确性
- 测试边界情况（负坐标、零坐标）

**GroupSizeAdjuster**：
- 测试群组扩大和缩小
- 测试 padding 计算
- 测试嵌套群组调整

**CollisionResolver**：
- 测试碰撞检测的准确性
- 测试碰撞解决的收敛性
- 测试边界情况（节点完全重叠）

**NestingTreeBuilder**：
- 测试深度计算的正确性
- 测试循环引用处理
- 测试多层嵌套
