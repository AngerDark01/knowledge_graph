# 布局系统重构 - 总体架构

## 📋 文档说明

本文档详细说明重构后的布局系统架构设计，基于真实源码实现编写。

**源码位置**：`src/services/layout/`

**最后更新**：2025-11-24

---

## 🎯 重构目标

### 重构前的问题

1. **单一巨型文件**：GridCenterLayoutStrategy.ts 包含 1359 行代码
2. **职责混乱**：一个类包含多种布局逻辑、工具方法、算法实现
3. **难以测试**：逻辑耦合严重，无法独立测试各个部分
4. **难以扩展**：添加新算法需要修改核心类
5. **存在死代码**：calculateGroupGridLayout() 方法从未被调用

### 重构后的改进

1. **职责分离**：工具层、算法层、策略层三层架构
2. **单一职责**：每个类只负责一个明确的功能
3. **依赖注入**：通过构造函数注入依赖，易于测试
4. **策略模式**：算法可插拔，易于扩展
5. **智能调度**：LayoutManager 自动选择最佳策略

## 📦 模块组成

### 1. 调度层（1个类）

**LayoutManager** - 布局管理器
- **文件**：`src/services/layout/LayoutManager.ts`
- **行数**：约 250 行
- **职责**：
  - 注册和管理所有布局策略
  - 根据选项智能选择策略
  - 协调布局操作的执行
  - 管理操作生命周期（取消、进度等）

### 2. 策略层（3个类）

**CanvasLayoutStrategy** - 画布布局策略
- **文件**：`src/services/layout/strategies/CanvasLayoutStrategy.ts`
- **行数**：约 150 行
- **职责**：布局所有顶层节点（无 groupId）

**GroupLayoutStrategy** - 群组布局策略
- **文件**：`src/services/layout/strategies/GroupLayoutStrategy.ts`
- **行数**：约 180 行
- **职责**：布局指定群组的直接子节点

**RecursiveLayoutStrategy** - 递归布局策略
- **文件**：`src/services/layout/strategies/RecursiveLayoutStrategy.ts`
- **行数**：约 200 行
- **职责**：从最深层到顶层递归布局所有嵌套结构

### 3. 算法层（3个类）

**ILayoutAlgorithm** - 算法接口
- **文件**：`src/services/layout/algorithms/ILayoutAlgorithm.ts`
- **行数**：约 70 行
- **职责**：定义算法标准接口

**GridAlgorithm** - 网格布局算法
- **文件**：`src/services/layout/algorithms/GridAlgorithm.ts`
- **行数**：约 160 行
- **职责**：从左上角 (0,0) 开始的网格布局

**GridCenterAlgorithm** - 居中网格算法
- **文件**：`src/services/layout/algorithms/GridCenterAlgorithm.ts`
- **行数**：约 170 行
- **职责**：以原点为中心的网格布局

**EdgeOptimizer** - 边优化算法
- **文件**：`src/services/layout/algorithms/EdgeOptimizer.ts`
- **职责**：优化边的连接点位置

### 4. 工具层（6个类）

**CoordinateTransformer** - 坐标转换工具
- **文件**：`src/services/layout/utils/CoordinateTransformer.ts`
- **行数**：约 140 行
- **职责**：处理所有坐标系转换

**GroupSizeAdjuster** - 群组大小调整工具
- **文件**：`src/services/layout/utils/GroupSizeAdjuster.ts`
- **行数**：约 200 行
- **职责**：自动调整群组大小以适应子节点

**CollisionResolver** - 碰撞解决工具
- **文件**：`src/services/layout/utils/CollisionResolver.ts`
- **行数**：约 170 行
- **职责**：检测和解决节点重叠

**NestingTreeBuilder** - 嵌套树构建器
- **文件**：`src/services/layout/utils/NestingTreeBuilder.ts`
- **职责**：构建和分析节点嵌套层级关系

**GeometryUtils** - 几何计算工具
- **文件**：`src/services/layout/utils/GeometryUtils.ts`
- **职责**：提供边界、距离等几何计算

**NestedNodePositionUpdater** - 嵌套节点位置更新器
- **文件**：`src/services/layout/utils/NestedNodePositionUpdater.ts`
- **职责**：根据父节点变化更新子节点位置

## 🔄 数据流转

### 典型调用流程

```
用户操作（点击布局按钮）
    │
    ▼
LayoutControl.tsx
    │
    ├─ 创建 LayoutManager 实例
    ├─ 调用 applyLayout(nodes, edges, options)
    │
    ▼
LayoutManager
    │
    ├─ 智能策略选择
    │   ├─ 检查 options.strategy（显式指定）
    │   ├─ 检查 options.layoutMode === 'recursive'
    │   ├─ 检查 options.targetGroupId（群组布局）
    │   └─ 默认使用 canvas-layout
    │
    ├─ 获取策略实例
    │
    ▼
选中的策略（Canvas/Group/Recursive）
    │
    ├─ 调用算法层计算布局
    ├─ 使用工具层处理坐标、碰撞等
    ├─ 优化边连接点
    │
    ▼
返回 LayoutResult
    │
    ├─ nodes: Map<nodeId, position>
    ├─ edges: Map<edgeId, handles>
    ├─ stats: { duration, iterations, collisions }
    │
    ▼
LayoutControl.tsx
    │
    ├─ 更新节点位置到 store
    ├─ 更新边连接点到 store
    └─ 触发 UI 重新渲染
```

## 🎨 设计模式应用

### 1. 策略模式（Strategy Pattern）

**定义**：定义一系列算法，把它们封装起来，并使它们可以互相替换。

**应用**：
- **抽象策略**：ILayoutStrategy 接口
- **具体策略**：CanvasLayoutStrategy、GroupLayoutStrategy、RecursiveLayoutStrategy
- **上下文**：LayoutManager

**好处**：
- 新增布局策略无需修改 LayoutManager
- 策略之间相互独立，易于测试
- 运行时可以动态切换策略

### 2. 依赖注入（Dependency Injection）

**定义**：通过构造函数或方法参数传入依赖，而非在类内部创建。

**应用实例**：

```typescript
// CanvasLayoutStrategy 的构造函数
constructor(
  algorithm?: ILayoutAlgorithm,           // 注入算法
  collisionResolver?: CollisionResolver,  // 注入工具
  edgeOptimizer?: EdgeOptimizer          // 注入工具
)
```

**好处**：
- 易于单元测试（可以注入 mock 对象）
- 降低耦合度
- 提高代码复用性

### 3. 组合模式（Composite Pattern）

**定义**：将对象组合成树形结构，使客户端对单个对象和组合对象的使用具有一致性。

**应用**：
- RecursiveLayoutStrategy 组合了 GroupLayoutStrategy 和 CanvasLayoutStrategy
- 递归处理嵌套群组的树形结构

**好处**：
- 统一处理单个节点和群组节点
- 递归处理更加自然

## 🔧 关键技术决策

### 1. 为什么分三层？

**工具层**：纯函数式工具，无副作用，可独立测试
**算法层**：只负责计算位置，不依赖具体业务
**策略层**：组合工具和算法，实现具体业务逻辑

**原因**：清晰的分层便于理解、测试和维护

### 2. 为什么使用依赖注入？

**测试友好**：可以注入 mock 对象进行单元测试
**灵活配置**：不同策略可以使用不同的算法
**降低耦合**：策略不直接依赖具体实现

### 3. 为什么保留旧策略？

**向后兼容**：避免破坏现有 API 调用
**平滑迁移**：给用户时间迁移到新策略
**风险控制**：新策略如有问题可以回退

### 4. 为什么使用智能选择？

**简化调用**：调用方无需关心策略细节
**减少错误**：避免手动选择错误的策略
**提升体验**：自动选择最佳策略
