# 布局系统重构 - 策略层详解

## 📋 文档说明

本文档详细说明策略层的三个布局策略，基于真实源码编写。

**源码位置**：`src/services/layout/strategies/`

**最后更新**：2025-11-24

---

## 🎯 策略层职责

策略层是布局系统的核心业务逻辑层，具有以下特点：

1. **实现 ILayoutStrategy 接口**：统一的策略接口
2. **组合算法和工具**：使用算法层和工具层完成布局
3. **处理业务逻辑**：理解节点类型、群组关系等
4. **返回布局结果**：包含位置、边、统计信息

---

## 🏗️ 策略接口设计

### ILayoutStrategy 接口

**定义**：
```typescript
interface ILayoutStrategy {
  readonly name: string           // 策略名称
  readonly id: string             // 策略唯一标识

  applyLayout(
    nodes,                        // 所有节点
    edges,                        // 所有边
    options?                      // 布局选项
  ): Promise<LayoutResult>        // 异步返回结果

  validateConfig?(config): boolean  // 可选：验证配置
}
```

**LayoutResult 结构**：
```typescript
interface LayoutResult {
  success: boolean                     // 是否成功
  nodes: Map<string, Position>         // 节点位置 Map
  edges: Map<string, OptimizedEdge>    // 优化后的边
  errors: string[]                     // 错误信息
  stats: {
    duration: number                   // 耗时（毫秒）
    iterations: number                 // 迭代次数
    collisions: number                 // 碰撞次数
  }
}
```

## 1️⃣ GroupLayoutStrategy - 群组布局策略

### 核心定位

**文件**：`src/services/layout/strategies/GroupLayoutStrategy.ts` (约180行)

**职责**：对指定群组的**直接子节点**进行布局

**关键特点**：
- 使用固定锚点（群组左上角 + padding）
- 自动调整群组大小
- 支持子节点去重
- 递归更新嵌套子节点位置

### 依赖注入

**构造函数**：
```typescript
constructor(
  algorithm?: ILayoutAlgorithm,        // 默认 GridAlgorithm
  collisionResolver?: CollisionResolver, // 默认 new CollisionResolver()
  nestedNodeUpdater?: NestedNodePositionUpdater // 默认 new NestedNodePositionUpdater()
)
```

**默认配置**：
- 算法：GridAlgorithm（从左上角开始）
- 碰撞解决：CollisionResolver
- 嵌套节点更新：NestedNodePositionUpdater

---

### 执行流程详解

#### 阶段1：参数验证

**检查 targetGroupId**：
```
if (!options?.targetGroupId) {
  throw Error('需要 targetGroupId')
}
```

**查找目标群组**：
```
parentGroup = nodes.find(n =>
  n.id === targetGroupId &&
  n.type === 'group'
)

if (!parentGroup) {
  throw Error('群组不存在')
}
```

---

#### 阶段2：获取直接子节点（去重）

**筛选子节点**：
```
children = nodes.filter(n =>
  'groupId' in n &&
  n.groupId === targetGroupId
)
```

**去重处理**：
```
使用 Map 去重：
uniqueMap = new Map()
for (child of children) {
  if (!uniqueMap.has(child.id)) {
    uniqueMap.set(child.id, child)
  }
}
children = Array.from(uniqueMap.values())
```

**为什么需要去重？**
- 防止数据异常导致重复节点
- 确保每个节点只布局一次
- 避免位置计算错误

**去重效果**：
```
去重前：[NodeA, NodeB, NodeA, NodeC]  // NodeA重复
去重后：[NodeA, NodeB, NodeC]         // 去重成功
```

---

#### 阶段3：使用算法计算相对布局

**调用 GridAlgorithm**：
```
relativeLayout = this.algorithm.calculate(children, {
  horizontalSpacing: options?.gridSpacing,
  verticalSpacing: options?.gridSpacing
})
```

**得到相对坐标**：
```
例如：
NodeA: (60, 40)     // 相对于 (0,0)
NodeB: (330, 40)
NodeC: (60, 220)
```

**特点**：
- 所有坐标都是正数（第四象限）
- 从 (0,0) 开始布局
- 不考虑群组实际位置

---

#### 阶段4：转换为绝对坐标

**计算群组锚点**：
```
使用 CoordinateTransformer.relativeToAbsoluteForGroup()

锚点 = 群组位置 + padding
例如：
  群组位置：(500, 300)
  padding：{ left: 20, top: 20 }
  锚点：(520, 320)
```

**转换坐标**：
```
绝对位置 = 相对位置 + 锚点

例如：
  NodeA: (60, 40) + (520, 320) = (580, 360)
  NodeB: (330, 40) + (520, 320) = (850, 360)
  NodeC: (60, 220) + (520, 320) = (580, 540)
```

**固定锚点的意义**：
- 锚点不会漂移
- 子节点位置稳定
- 多次布局结果一致

---

#### 阶段5：碰撞检测与解决

**检测碰撞**：
```
collisionCount = this.collisionResolver.countCollisions(absoluteLayout)
```

**解决碰撞**：
```
if (collisionCount > 0) {
  resolvedLayout = this.collisionResolver.resolve(absoluteLayout)
  console.log(`解决了 ${collisionCount} 个碰撞`)
}
```

**碰撞解决原理**：
- 使用最小平移向量（MTV）
- 迭代推开重叠节点
- 最多迭代 50 次

---

#### 阶段6：递归更新嵌套子节点位置

**调用工具更新**：
```
allUpdatedNodes = NestedNodePositionUpdater.updateNestedNodePositionsForGroupLayout(
  nodes,
  children,
  resolvedLayout
)
```

**更新内容**：
- 直接子节点的新位置
- 所有嵌套在子节点内的节点位置
- 保持相对位置不变

**示例**：
```
群组布局前：
  GroupX: (100, 100)
    └─ NodeA: (120, 120)  // 相对GroupX: (20, 20)
        └─ NodeB: (140, 140)  // 相对NodeA: (20, 20)

群组布局后：
  GroupX: (500, 300)  // 新位置
    └─ NodeA: (580, 360)  // 新位置
        └─ NodeB: (600, 380)  // 跟随NodeA移动

计算：
  NodeA移动量: (580-120, 360-120) = (460, 240)
  NodeB新位置: (140+460, 140+240) = (600, 380)
```

---

#### 阶段7：自适应调整群组大小

**调用工具调整**：
```
adjustedNodes = GroupSizeAdjuster.adjustSingleGroup(
  targetGroupId,
  [...nodes]  // 传入所有节点的副本
)
```

**调整过程**：
1. 计算子节点相对边界
2. 计算需要的群组尺寸
3. 更新群组 width、height、boundary

**调整效果**：
```
调整前：
  群组尺寸：300 x 200
  子节点超出边界

调整后：
  群组尺寸：450 x 350
  完整包含所有子节点
```

---

#### 阶段8：构建返回结果

**创建位置 Map**：
```
nodePositions = new Map()
for (node of allUpdatedNodes) {
  nodePositions.set(node.id, node.position)
}
```

**添加群组尺寸信息**：
```
updatedGroup = adjustedNodes.find(n => n.id === targetGroupId)
groupPos = nodePositions.get(targetGroupId)
if (groupPos) {
  groupPos.width = updatedGroup.width
  groupPos.height = updatedGroup.height
  groupPos.boundary = updatedGroup.boundary
}
```

**返回结果**：
```typescript
return {
  success: true,
  nodes: nodePositions,      // 包含子节点位置和群组尺寸
  edges: new Map(),          // 群组布局不处理边
  errors: [],
  stats: {
    duration,
    iterations: 1,
    collisions: collisionCount
  }
}
```

---

### 使用场景

**适用于**：
- ✅ 用户选中群组，点击"布局群组内部"
- ✅ RecursiveLayoutStrategy 调用
- ✅ 需要布局特定群组的子节点

**不适用于**：
- ❌ 顶层节点布局
- ❌ 递归布局所有群组
- ❌ 画布整体布局

---

### 关键设计决策

#### 为什么使用固定锚点？

**对比**：

**漂移锚点（中心点）**：
```
第1次布局：锚点在群组中心
  ↓ 子节点分布
  ↓ 群组扩大
第2次布局：锚点位置改变（中心漂移）
  ↓ 子节点位置全部改变
  ↓ 视觉不稳定
```

**固定锚点（左上角+padding）**：
```
第1次布局：锚点在左上角+padding
  ↓ 子节点分布
  ↓ 群组扩大
第2次布局：锚点位置不变
  ↓ 子节点位置稳定
  ↓ 视觉稳定
```

#### 为什么自动调整群组大小？

**用户体验**：
- 用户不需要手动调整群组大小
- 群组总是能完整包含子节点
- 避免子节点显示不全

#### 为什么递归更新嵌套子节点？

**多层嵌套支持**：
- Group → Group → Node 的嵌套结构
- 保持所有层级的相对位置不变
- 确保布局的一致性

## 2️⃣ CanvasLayoutStrategy - 画布布局策略

### 核心定位

**文件**：`src/services/layout/strategies/CanvasLayoutStrategy.ts` (约150行)

**职责**：对所有**顶层节点**（无 groupId）进行布局

**关键特点**：
- 使用居中布局算法
- 支持权重排序
- 自动更新嵌套节点位置
- 优化所有边的连接点

### 依赖注入

**构造函数**：
```typescript
constructor(
  algorithm?: ILayoutAlgorithm,         // 默认 GridCenterAlgorithm
  collisionResolver?: CollisionResolver,
  edgeOptimizer?: EdgeOptimizer,
  nestedNodeUpdater?: NestedNodePositionUpdater
)
```

**默认配置**：
- 算法：GridCenterAlgorithm（居中布局）
- 碰撞解决：CollisionResolver
- 边优化：EdgeOptimizer
- 嵌套节点更新：NestedNodePositionUpdater

---

### 执行流程详解

#### 阶段1：筛选顶层节点

**筛选条件**：
```
topLevelNodes = nodes.filter(node =>
  !('groupId' in node) ||
  !node.groupId
)
```

**顶层节点定义**：
- 没有 groupId 属性的节点
- groupId 为 null 或 undefined 的节点

**示例**：
```
所有节点：
  NodeA (无 groupId)          -> 顶层 ✅
  NodeB (groupId: null)       -> 顶层 ✅
  GroupX (无 groupId)         -> 顶层 ✅
  NodeC (groupId: 'GroupX')   -> 嵌套 ❌
  NodeD (groupId: 'GroupX')   -> 嵌套 ❌

顶层节点：[NodeA, NodeB, GroupX]
```

---

#### 阶段2：计算节点权重（可选）

**触发条件**：
```
if (options?.useWeightedLayout) {
  nodeWeights = this.calculateWeights(topLevelNodes, edges)
}
```

**权重计算公式**：
```
对于每个节点：
  1. 计算面积权重
     area = width * height
     areaWeight = min(1, area / maxArea)

  2. 计算边数权重
     totalEdgeWeight = min(1, edgeCount / maxEdges)
     inEdgeWeight = min(1, inEdges / maxEdges)
     outEdgeWeight = min(1, outEdges / maxEdges)

  3. 综合权重
     totalWeight = areaWeight * 0.2
                 + totalEdgeWeight * 0.3
                 + inEdgeWeight * 0.25
                 + outEdgeWeight * 0.25
```

**权重配置**：
```
maxArea = 100000     // 最大面积
maxEdges = 10        // 最大边数
```

**权重示例**：
```
NodeA:
  面积：120 * 80 = 9600
  总边数：5
  入边：3
  出边：2

  areaWeight = min(1, 9600/100000) = 0.096
  totalEdgeWeight = min(1, 5/10) = 0.5
  inEdgeWeight = min(1, 3/10) = 0.3
  outEdgeWeight = min(1, 2/10) = 0.2

  totalWeight = 0.096*0.2 + 0.5*0.3 + 0.3*0.25 + 0.2*0.25
              = 0.0192 + 0.15 + 0.075 + 0.05
              = 0.2942
```

---

#### 阶段3：使用算法计算居中布局

**调用 GridCenterAlgorithm**：
```
layoutedNodes = this.algorithm.calculate(topLevelNodes, {
  horizontalSpacing: options?.gridSpacing || 默认值,
  verticalSpacing: options?.gridSpacing || 默认值,
  useWeightedOrder: options?.useWeightedLayout,
  nodeWeights
})
```

**得到居中坐标**：
```
例如（3个节点）：
NodeA: (-270, -90)    // 左侧
NodeB: (0, -90)       // 中间（权重最高）
GroupX: (270, -90)    // 右侧
```

**特点**：
- 坐标可能包含负数
- 整体居中在原点
- 高权重节点靠近中心

---

#### 阶段4：碰撞检测与解决

**与 GroupLayoutStrategy 相同**：
```
resolvedNodes = this.collisionResolver.resolve(layoutedNodes)
collisionCount = this.collisionResolver.countCollisions(layoutedNodes)
```

---

#### 阶段5：更新嵌套节点位置

**关键步骤**：更新所有嵌套节点的绝对位置

**原理**：
- 顶层节点位置改变
- 嵌套节点需要跟随移动
- 保持相对位置不变

**处理逻辑**：

**步骤1：创建节点映射**
```
originalNodeMap = Map(原始节点)
layoutedNodeMap = Map(布局后顶层节点)
```

**步骤2：处理顶层节点**
```
resultNodes = [...resolvedNodes]
```

**步骤3：处理嵌套节点**
```
nestedNodes = 所有有 groupId 的节点

for (nestedNode of nestedNodes) {
  absolutePosition = calculateAbsolutePosition(
    nestedNode,
    originalNodeMap,
    layoutedNodeMap
  )
  resultNodes.push({
    ...nestedNode,
    position: absolutePosition
  })
}
```

**步骤4：递归计算绝对位置**
```
function calculateAbsolutePosition(node):
  if (node 无 groupId):
    return layoutedNodeMap.get(node.id).position

  originalParentGroup = originalNodeMap.get(node.groupId)
  layoutedParentGroup = layoutedNodeMap.get(node.groupId)

  // 计算相对位置
  relativeX = node.x - originalParentGroup.x
  relativeY = node.y - originalParentGroup.y

  if (layoutedParentGroup 存在):
    // 父群组已布局，直接使用新位置
    return {
      x: layoutedParentGroup.x + relativeX,
      y: layoutedParentGroup.y + relativeY
    }
  else:
    // 父群组是嵌套群组，递归计算
    parentAbsPos = calculateAbsolutePosition(originalParentGroup)
    return {
      x: parentAbsPos.x + relativeX,
      y: parentAbsPos.y + relativeY
    }
```

**示例**：
```
布局前：
  GroupX: (100, 100)
    └─ NodeA: (120, 120)  相对位置: (20, 20)

布局后：
  GroupX: (0, -90)  新位置
    └─ NodeA: ?

计算：
  NodeA.x = 0 + 20 = 20
  NodeA.y = -90 + 20 = -70
  NodeA: (20, -70)
```

**多层嵌套示例**：
```
布局前：
  GroupA: (100, 100)
    └─ GroupB: (120, 120)  相对GroupA: (20, 20)
        └─ NodeC: (140, 140)  相对GroupB: (20, 20)

布局后：
  GroupA: (0, 0)  新位置

计算：
  GroupB位置 = (0, 0) + (20, 20) = (20, 20)
  NodeC位置 = (20, 20) + (20, 20) = (40, 40)
```

---

#### 阶段6：优化边的连接点

**调用边优化器**：
```
optimizedEdges = this.edgeOptimizer.optimizeEdgeHandles(
  resultNodes,  // 所有节点（含嵌套节点）
  edges
)
```

**优化效果**：
- 选择最优连接点
- 减少边的长度
- 减少边的交叉

---

#### 阶段7：构建返回结果

**创建节点位置 Map**：
```
nodePositions = new Map()
for (node of resultNodes) {
  nodePositions.set(node.id, node.position)
}
```

**创建边 Map**：
```
edgeHandles = new Map()
for (edge of optimizedEdges) {
  edgeHandles.set(edge.id, edge)
}
```

**返回结果**：
```typescript
return {
  success: true,
  nodes: nodePositions,      // 所有节点位置（含嵌套）
  edges: edgeHandles,        // 优化后的边
  errors: [],
  stats: {
    duration,
    iterations: 1,
    collisions: collisionCount
  }
}
```

### 使用场景

**适用于**：
- ✅ 用户点击"应用布局"按钮
- ✅ 布局所有顶层节点
- ✅ RecursiveLayoutStrategy 最后阶段

**不适用于**：
- ❌ 布局特定群组内部
- ❌ 只布局部分节点

---

### 关键设计决策

#### 为什么使用居中布局？

**用户体验**：
- 画布中心是视觉焦点
- 居中布局更美观
- 避免节点都在角落

#### 为什么支持权重排序？

**视觉突出**：
- 重要节点（高权重）更靠近中心
- 视觉焦点更清晰
- 符合用户直觉

#### 为什么更新嵌套节点位置？

**保持一致性**：
- 父节点移动，子节点必须跟随
- 保持相对位置不变
- 避免子节点"遗落"

## 3️⃣ RecursiveLayoutStrategy - 递归布局策略

### 核心定位

**文件**：`src/services/layout/strategies/RecursiveLayoutStrategy.ts` (约200行)

**职责**：递归布局**所有嵌套层级**的节点

**关键特点**：
- 从最深层到顶层逐层布局
- 编排 GroupLayoutStrategy 和 CanvasLayoutStrategy
- 支持进度回调
- 自动调整所有群组大小

### 依赖注入

**构造函数**：
```typescript
constructor(
  groupStrategy?: GroupLayoutStrategy,
  canvasStrategy?: CanvasLayoutStrategy,
  treeBuilder?: NestingTreeBuilder,
  edgeOptimizer?: EdgeOptimizer,
  nestedNodeUpdater?: NestedNodePositionUpdater
)
```

**依赖关系**：
- groupStrategy：布局每个群组
- canvasStrategy：布局顶层节点
- treeBuilder：分析嵌套结构
- edgeOptimizer：优化所有边
- nestedNodeUpdater：更新嵌套节点位置

---

### 执行流程详解

#### 阶段1：构建嵌套层级树

**调用工具构建**：
```
depthGroups = this.treeBuilder.groupNodesByDepth(nodes)
maxDepth = this.treeBuilder.getMaxDepth(depthGroups)
```

**分组结果示例**：
```
depthGroups = {
  0: [GroupA, GroupC, NodeX],        // 顶层
  1: [NodeA1, GroupB, NodeC1],       // 第1层
  2: [NodeB1, NodeB2]                // 第2层
}
maxDepth = 2
```

**报告日志**：
```
console.log(`检测到 ${depthGroups.size} 层嵌套结构，最大深度: ${maxDepth}`)
```

---

#### 阶段2：创建工作节点副本

**为什么需要副本？**
- 逐步更新节点位置
- 每次布局使用最新位置
- 不修改原始输入

**创建副本**：
```
workingNodes = nodes.map(n => ({
  ...n,
  position: { ...n.position }
}))
```

---

#### 阶段3：从最深层向上逐层布局群组

**循环结构**：
```
for (depth = maxDepth; depth > 0; depth--) {
  groupsAtDepth = getGroupsAtDepth(depthGroups, depth)

  for (group of groupsAtDepth) {
    布局该群组的子节点
  }
}
```

**为什么从深到浅？**
- 深层子节点先布局好
- 浅层群组大小才能正确计算
- 自下而上确保一致性

**处理每个群组**：

**步骤1：检查子节点**
```
childrenCount = workingNodes.filter(
  n => n.groupId === group.id
).length

if (childrenCount === 0) {
  console.log(`群组 ${group.id} 没有子节点，跳过`)
  continue
}
```

**步骤2：调用群组布局策略**
```
layoutResult = await this.groupStrategy.applyLayout(
  workingNodes,
  edges,
  {
    targetGroupId: group.id,
    gridSpacing: options?.gridSpacing,
    ...options
  }
)
```

**步骤3：合并布局结果**
```
if (layoutResult.success) {
  mergeLayoutResult(workingNodes, allUpdatedPositions, layoutResult)
  stats.iterations += layoutResult.stats.iterations
  stats.collisions += layoutResult.stats.collisions
  totalProcessedNodes += layoutResult.nodes.size
}
```

**步骤4：发送进度更新**
```
if (options?.onProgress) {
  options.onProgress({
    currentLevel: maxDepth - depth + 1,
    totalLevels: maxDepth + 1,
    processedNodes: totalProcessedNodes,
    totalNodes: totalNodes
  })
}
```

#### 阶段5：从深到浅调整所有群组大小

**调整流程**：
```
adjustedNodes = this.adjustAllGroupSizes(
  workingNodes,
  depthGroups,
  maxDepth
)
```

**调整逻辑**：
```
for (depth = maxDepth; depth >= 0; depth--) {
  groupsAtDepth = getGroupsAtDepth(depthGroups, depth)

  for (group of groupsAtDepth) {
    children = workingNodes.filter(
      n => n.groupId === group.id
    )

    if (children.length === 0) continue

    workingNodes = GroupSizeAdjuster.adjustSingleGroup(
      group.id,
      workingNodes
    )
  }
}
```

**为什么再次从深到浅？**
- 深层群组大小先调整
- 浅层群组才能正确包含子群组
- 确保所有群组都合适

### 关键设计决策

#### 为什么是编排器而非实现者？

**设计理念**：
- 不重复实现布局逻辑
- 复用已有的群组和画布布局策略
- 专注于编排逻辑（先布局深层，再布局浅层）
- 保持策略的独立性

#### 为什么从深到浅布局？

**正确性保证**：
- 深层节点先固定位置
- 浅层群组大小基于深层节点调整
- 避免后续布局破坏已布局的节点

#### 为什么支持进度回调？

**用户体验**：
- 大图布局时提供进度反馈
- 用户可以感知操作在进行
- 适用于性能敏感场景

## 🔄 策略间协作

### 递归策略与子策略的协作

```
RecursiveLayoutStrategy
    ├── 对于深度2: 调用 GroupLayoutStrategy
    │   └── GroupLayoutStrategy 使用 GridAlgorithm 布局深度2的群组
    ├── 对于深度1: 调用 GroupLayoutStrategy  
    │   └── GroupLayoutStrategy 使用 GridAlgorithm 布局深度1的群组
    ├── 对于深度0: 调用 CanvasLayoutStrategy
    │   └── CanvasLayoutStrategy 使用 GridCenterAlgorithm 布局顶层节点
    └── 最后: 优化所有边（EdgeOptimizer）
```

### 数据传递流程

所有策略都使用相同的节点和边数据：

```
原始数据 (nodes, edges)
    ├── RecursiveLayoutStrategy  (全图)
    │    ├── GroupLayoutStrategy (深度2群组) → 局部结果
    │    ├── GroupLayoutStrategy (深度1群组) → 局部结果  
    │    └── CanvasLayoutStrategy (顶层节点) → 局部结果
    │         └── EdgeOptimizer (所有边) → 边结果
    └── 返回完整布局结果
```

## 🎯 策略选择指南

### 何时使用 GroupLayoutStrategy？

**场景**：
- 用户选中群组并点击"布局群组内部"
- 需要重新排列特定群组的子节点
- 布局群组内嵌套结构

**特点**：
- 针对特定群组
- 从左上角开始布局
- 保持群组边界

### 何时使用 CanvasLayoutStrategy？

**场景**：
- 用户点击"应用布局"
- 布局画布上的所有顶层节点
- 需要居中效果

**特点**：
- 处理顶层节点
- 支持权重排序
- 优化全图边

### 何时使用 RecursiveLayoutStrategy？

**场景**：
- 用户点击"递归布局全部"
- 初始化整个图的布局
- 重构图结构后的全面布局

**特点**：
- 处理所有层级
- 从深到浅执行
- 保证布局一致性
