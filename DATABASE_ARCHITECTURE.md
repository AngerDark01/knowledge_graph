# 知识图谱项目数据库架构设计方案

## 项目概述

本方案针对知识图谱前端项目设计持久化数据库解决方案，主要解决两个核心需求：
1. 支持创建和管理多个画布
2. 实现画布状态的持久化保存和恢复

## 数据库架构总览

采用混合数据库架构，结合图数据库、文档数据库和缓存数据库的优势，构建高效、可扩展的知识图谱存储系统。

## 数据库选型及职责

### 1. Neo4j（图数据库）

**用途**：存储和管理图谱的拓扑结构

**具体职责**：
- 存储节点间的连接关系
- 管理嵌套群组结构（节点-群组、群组-群组关系）
- 维护画布的图拓扑结构
- 处理图遍历查询（如路径查找、邻居节点查询）
- 管理节点的父子关系（嵌套结构）
- 存储边的属性和关系类型
- 支持RDF语义关系的图结构表示

**优势**：
- 优化图遍历操作，查询性能优异
- 天然支持复杂的图结构关系
- 支持ACID事务，数据一致性有保障
- 适合存储知识图谱的连接性数据

### 2. Elasticsearch（搜索引擎）

**用途**：存储节点的详细内容和文档数据

**具体职责**：
- 存储节点的详细文本内容
- 管理笔记节点的内容数据
- 提供全文搜索功能
- 存储RDF属性数据和元数据
- 处理复杂的内容过滤和排序
- 支持多字段搜索和相关性分析
- 管理标签、分类等结构化属性
- 提供内容的高亮显示功能

**优势**：
- 强大的全文搜索能力
- 高效的文档存储和检索
- 支持复杂的查询和聚合
- 水平扩展能力强
- 适合存储和检索大量文本内容

### 3. Redis（内存数据库）

**用途**：高速缓存和会话管理

**具体职责**：
- 缓存热数据，提升访问性能
- 管理用户会话和认证信息
- 存储实时协作的临时状态
- 实现分布式锁机制
- 处理消息队列（用于实时协作）
- 缓存频繁查询的图结构数据
- 存储画布的实时编辑状态

**优势**：
- 极高的读写性能
- 支持多种数据结构
- 内置过期策略，自动清理临时数据
- 支持发布/订阅模式，适合实时应用

## 架构设计

### 整体架构图

```
                    ┌─────────────────┐
                    │   客户端应用     │
                    └─────────┬───────┘
                              │
                    ┌─────────▼───────┐
                    │    API 层       │
                    └─────────┬───────┘
            ┌─────────────────┼─────────────────┐
            │                 │                 │
    ┌───────▼──────┐  ┌───────▼──────┐  ┌───────▼──────┐
    │  Neo4j       │  │ Elasticsearch│  │ Redis        │
    │ (图结构)      │  │ (内容数据)    │  │ (缓存)      │
    └──────────────┘  └──────────────┘  └──────────────┘
```

### 数据流向设计

#### 读取流程
1. 用户请求画布数据
2. 服务优先查询Redis缓存，若命中则直接返回
3. 缓存未命中时，同时查询Neo4j获取拓扑结构
4. 并行查询Elasticsearch获取节点内容
5. 整合数据后返回给前端
6. 将结果缓存到Redis

#### 写入流程
1. 用户修改画布数据
2. 同时写入Neo4j（拓扑关系）和Elasticsearch（内容）
3. 更新Redis中的缓存数据
4. 发送变更通知到协作用户

### 服务层设计

#### 图结构服务
- 负责管理Neo4j中的图拓扑数据
- 处理节点连接关系操作
- 执行图遍历算法

#### 内容服务
- 负责管理Elasticsearch中的文档数据
- 提供全文搜索功能
- 处理RDF属性管理

#### 缓存服务
- 负责Redis缓存的管理
- 实现数据一致性保障
- 处理会话管理

## 数据库部署策略

### 高可用部署
- **Neo4j**: 集群部署，支持故障自动切换
- **Elasticsearch**: 分片副本机制，确保数据安全
- **Redis**: 主从配置+哨兵模式，确保服务不中断

### 性能优化
- **分片策略**: 按画布ID对数据进行分片
- **索引优化**: 在Neo4j和Elasticsearch中建立合适索引
- **缓存策略**: 合理设置Redis缓存过期时间

## 协作机制

### 实时协作支持
- Redis存储协作状态
- 通过Elasticsearch的实时索引支持内容搜索
- Neo4j维护正确的图结构关系

### 数据一致性
- 采用最终一致性模型
- 通过消息队列确保数据同步
- 实现冲突检测和解决机制

## 扩展性设计

### 水平扩展
- Neo4j集群支持大规模图数据
- Elasticsearch分片机制支持海量内容
- Redis集群支持高并发访问

### 功能扩展
- 支持更多RDF语义功能
- 可扩展全文搜索能力
- 支持复杂图分析算法

## 安全设计

### 数据安全
- 数据库访问权限控制
- 敏感数据加密存储
- 网络通信加密

### 应用安全
- API访问认证和授权
- 查询注入防护
- 访问频率限制

## 总结

该混合数据库架构充分利用了不同数据库类型的优势：

- **Neo4j** 负责高效的图结构操作和复杂关系查询
- **Elasticsearch** 负责强大的内容检索和RDF属性管理
- **Redis** 负责高速缓存和实时协作状态管理

这种架构设计能够完美支持知识图谱的多画布管理和状态持久化需求，同时具备良好的性能、扩展性和可靠性。
