# Ontology Canvas / Knowledge Graph Editor

## 产品 PRD（第一版）

基于当前概念图体系整理。
目标：构建一个支持「本体建模 + 节点嵌套 + 超大图浏览 + 多层子画布」的 Ontology IDE，而不是传统白板。

---

# 1. 产品定位

Ontology Canvas 是一个：

```text
结构化知识建模系统
+
图结构 IDE
+
可嵌套的本体画布系统
```

不是：

* 普通白板
* 思维导图
* 流程图工具

核心目标：

```text
让用户可以在超大型知识结构中：
建模
组织
导航
聚焦
折叠
分层
引用
浏览
```

---

# 2. 核心设计理念

---

## 2.1 节点不是卡片

节点是：

```text
结构化本体实体
```

节点可以拥有：

* 属性
* 方法
* 关系
* 子节点
* 子画布
* 继承结构
* 引用关系

节点本身就是：

```text
一个局部知识空间
```

---

## 2.2 节点内部是“子画布”

核心理念：

```text
节点展开后，
内部不是字段列表，
而是一个独立画布空间。
```

例如：

```text
Order
  └── 内部画布
        ├── Customer
        ├── Payment
        └── Product
```

节点本质：

```text
Node = Container + Viewport + Semantic Entity
```

---

## 2.3 不存在独立 Domain

当前方案取消：

```text
Domain 作为独立实体
```

原因：

* 与节点嵌套系统重复
* 概念层级冗余
* 用户认知负担高

替代方案：

```text
任意节点都可成为“容器节点”
```

即：

```text
节点展开
=
进入内部子画布
```

---

# 3. 节点系统（Node System）

---

# 3.1 节点类型

当前支持：

| 类型       | 用途       |
| -------- | -------- |
| Class    | 本体类      |
| Concept  | 抽象概念     |
| Method   | 行为节点     |
| Property | 属性节点     |
| Service  | 服务       |
| Module   | 模块       |
| Event    | 事件       |
| Workflow | 工作流      |
| Agent    | AI Agent |

后续支持自定义类型。

---

# 3.2 节点状态

节点支持多级状态：

| 状态    | 描述      |
| ----- | ------- |
| 完整模式  | 展示完整内容  |
| 摘要模式  | 仅展示核心字段 |
| 最小模式  | 仅标题     |
| LOD模式 | 超远景聚合态  |

---

# 3.3 节点结构

节点基础结构：

```ts
Node {
  id
  title
  type

  fields[]
  methods[]

  relations[]

  childCanvasId

  visualState
}
```

---

# 4. 子画布系统（Sub Canvas System）

这是整个产品核心。

---

# 4.1 子画布概念

每个节点：

```text
都可以拥有自己的内部画布
```

即：

```text
Node
  -> SubCanvas
```

用户：

```text
双击节点
↓
进入内部子画布
```

---

# 4.2 子画布特性

子画布拥有：

* 独立节点
* 独立边
* 独立缩放
* 独立布局
* 独立LOD
* 独立视口状态

但：

```text
共享全局语义体系
```

---

# 4.3 子画布导航

支持：

| 功能         | 描述    |
| ---------- | ----- |
| 双击进入       | 进入子画布 |
| Breadcrumb | 层级导航  |
| 返回父级       | 回到上一层 |
| 历史导航       | 后退/前进 |
| 搜索跳转       | 自动定位  |

---

# 4.4 子画布 UI 规则（重要）

## 不采用：

```text
“每个子画布都拥有完整侧边栏”
```

原因：

* 信息重复
* UI 冗余
* 干扰聚焦
* 视觉层级混乱

---

## 正确方案：

### 子画布：

```text
只保留：
- 顶部导航栏
- Breadcrumb
- MiniMap
- 缩放工具
```

不重复：

* 属性面板
* 历史栏
* 工具栏
* 右侧编辑器

---

## 编辑器保持全局唯一

即：

```text
整个系统：
只有一个全局 Inspector
```

位于：

```text
主编辑器右侧
```

无论进入哪个子画布：

```text
右侧 Inspector 不变
```

这样：

* 状态一致
* 认知稳定
* 不会像 iframe 套 iframe

---

# 9. 当前设计原则

---

## 9.1 聚焦优先

进入子画布后：

```text
聚焦当前上下文
```

避免 UI 套娃。

---

## 9.2 节点 = 空间

节点不仅是数据。

还是：

```text
语义空间入口
```

---

## 9.3 浏览优先于编辑

这是：

```text
大型知识浏览系统
```

不是表单编辑器。

---

## 9.4 可无限嵌套

支持：

```text
Node
 -> SubCanvas
    -> Node
       -> SubCanvas
```

# 10. 当前落地优先级（2026-05-07）

当前开发不直接进入布局算法阶段，先修画布基础交互和 UI 表达。

## 10.1 第一优先级：交互正确与顺滑

必须先解决：

* 节点拖拽后不回弹。
* 容器节点移动时，内部节点保持相对位置。
* 节点 resize 不能一卡一卡。
* resize 过程中不写旧 graph store，不做全图布局，不做全图保存。
* resize 结束后只提交本体 view 尺寸，并按需更新容器边界。
* 节点移动或 resize 后，相关边锚点自动切换到更合适的位置。

## 10.2 第二优先级：节点 UI 与属性表达

节点 UI 参考当前提供的概念图，目标是类似 UML / IDE / 结构化类节点，而不是普通卡片。

节点至少要表达：

* 节点类型图标，例如 Class / Property / Method / Child Node。
* 标题和类型。
* Fields / Methods / Child Nodes 的数量。
* 完整、紧凑、最小、远景 LOD 状态。
* 选中、悬停、禁用、只读等状态。

所有尺寸、间距、字号、圆角、LOD 阈值必须进入配置或 token，不允许在组件中散落硬编码。

## 10.3 第三优先级：正交边系统

边系统目标不是普通曲线，而是正交边：

* Top / Right / Bottom / Left 通用锚点。
* Field / Method / Section 等语义锚点。
* 节点位置变化后，边自动选择更合适锚点。
* 大图远景下，边逐级简化、隐藏标签、聚合平行边。
* 关系类型通过线型、箭头、颜色、端点表达。

短期可以先保留 ReactFlow 边实现，但边锚点和语义关系必须写入本体 view，不能只写临时显示缓存。

## 10.4 第四优先级：子画布导航

子画布导航参考图中的系统设计：

* 双击或明确入口进入节点内部子画布。
* 顶部 Breadcrumb 表示路径。
* 返回父级时恢复父画布视口状态。
* 搜索可以定位到节点、属性、方法，并自动进入目标子画布。
* MiniMap 跟随当前子画布，提供大图定位。
